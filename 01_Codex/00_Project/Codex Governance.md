## Purpose

This document defines how the Alamat Codex is maintained as the official design documentation for *Alamat ni Jay Jay*.

The Codex should read like the internal documentation of a professional game studio and the curated record of an evolving RPG universe.

## Keeper Responsibilities

The Keeper of the Alamat Codex is responsible for:

- Maintaining canon.
- Tracking design decisions.
- Organizing documentation.
- Preventing contradictions.
- Recording project history.
- Managing document versions.
- Expanding existing documents.
- Suggesting future documentation.
- Preserving the identity, emotional authenticity, and continuity of Alamat.

## Canon Stewardship Rules

1. Never delete established canon.
2. Always build upon existing knowledge.
3. Preserve approved lore unless Jeffrey Moralde explicitly changes it.
4. When new ideas conflict with established lore, identify the conflict before making changes.
5. Draft ideas must remain clearly marked as draft until approved.
6. Deprecated material should be preserved for history, not silently erased.
7. Major creative changes must be reflected in the Decision Log.

## Documentation Standards

Every major Codex document should include:

- Clear title.
- Status.
- Canon Status.
- Last Updated date.
- Purpose or design intent.
- Organized sections that can expand over time.
- Open questions when decisions are unresolved.

## Version Management

Use simple document version labels:

| Version Type | Meaning |
| --- | --- |
| v0.x | Draft development. |
| v1.0 | First approved version. |
| v1.x | Approved revision or expansion. |
| v2.0 | Major approved restructuring or canon update. |

Version changes should reflect the document's maturity, not the number of edits.

## Contradiction Handling

When a possible contradiction appears:

1. Identify the conflicting documents or decisions.
2. Determine which material is approved, draft, deprecated, or internal.
3. Avoid overwriting approved canon.
4. Record the issue in `00_Project/Contradiction Register.md` if it cannot be resolved immediately.
5. Ask for explicit approval before changing established lore.

## Future Documentation Practice

When the Codex grows, new documents should be suggested when they reduce confusion, protect continuity, or make future design work easier.

Potential future documents include:

- Canon Glossary.
- Name Registry.
- Timeline of Approved Events.
- Location Index.
- Faction Index.
- Quest Index.
- Item Index.
- Bestiary Index.
- Cultural Authenticity Notes.
- Production Milestone Log.
